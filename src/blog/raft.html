come kill my cluster!

jul 18, 2026 · updated jul 21, 2026 · by anush sonone

this is a from-scratch raft key-value store you can break on purpose. seven dockerized machines replicate a log over grpc; a whitelist control plane lets you kill one and watch election, catch-up, and silent heal.

the dashboard below is the public surface. loadgen keeps real writes and linearizable reads moving on an oracle aarch64 vm (4 ocpu). prefer killing the leader. machines come back on their own (~10s). about one kill every 2 seconds per computer.

how reads stay cheap

early builds put every get through the raft log like a write. at high read qps that means thousands of proposals per second, one leader, six replicas, and disk syncs: heartbeats slip, elections storm, throughput collapses.

gets now use readindex (raft paper §6.4): the leader proves it still has a majority with a heartbeat round, waits until its local commit is applied, then serves the in-memory map. concurrent readers share one confirm; a short lease skips repeats. puts still go through the log, but concurrent puts group-commit (many entries, one fsync, one appendentries).

that is the difference between “turn up qps and hope” and a load the cluster can hold while you kill machines.

how we load-tested it

hud numbers are not marketing fiction. before raising public loadgen on the oracle host, we laddered setpoints after readindex landed. pass rule: raft term flat with no kills, loadgen errors near zero, some cpu left for chaos.

  1. prove log-replicated gets + 1.5k writes / 10k reads election-storm locally.
  2. ship readindex + lease + propose batching.
  3. rebuild oracle on clean volumes; smoke at 80/160 with one kill and heal.
  4. raise loadgen via a temporary override only; hold each step ~3 minutes.
  5. promote an honest rate under the measured peak (rollback: one git revert).

aspiration was 1.5k writes / 10k reads. on this 4 ocpu box that is too high: saturation sits around ~1.4k writes / ~3.2k reads. public loadgen is set to 1400 writes / 3000 reads so kills still have headroom. followers can lag by hundreds of commit indexes under load; that is normal catch-up, not “zero drift.”

c5 was the best sustained throughput. c6 asked for more and delivered less (cpu pegged). hud rates move with scrape windows and visitors; the table is the measured ceiling, not a guarantee every second matches those integers. code and compose: kill-my-cluster.

what this project is

kill-my-cluster is a custom raft kv in go: wal + snapshots, election and replication, replicated map with exactly-once client ids, whitelist kill/partition/heal + sse, prometheus, grafana, and loadgen.

edge is cloudflare tunnel; this page is the ui (vercel). there is no separate demo hostname. the teaching beat is still: kill the leader, watch the group elect and catch up, keep traffic honest under load.

learn raft

if raft is new, start with the secret lives of data, then in search of an understandable consensus algorithm and raft.github.io.

learn grpc

nodes talk with grpc over http/2 and protocol buffers. start with core concepts and the go quickstart.

how the repo maps

internal/storage wal and snapshots. internal/raft election, replication, readindex, propose batching. internal/kv replicated map + exactly-once client ids. internal/controlplane whitelist kill, partition, heal, sse. deploy/compose seven nodes + prometheus + grafana + loadgen.

local ports: control plane :8080, grafana :3001, prometheus :9090. public ui is this page only.

suggested path
